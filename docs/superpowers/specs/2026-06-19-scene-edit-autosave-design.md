# 場次編輯自動儲存與批次日期設計

日期：2026-06-19（Asia/Taipei）

## 目標

1. 編輯既有場次後，點擊頁面其他操作時，自動儲存目前內容。
2. 點擊另一場的「編輯」時，先完成目前場次儲存，再切換到下一場。
3. 批次修改日期時，日期欄預設為今日。
4. 電腦版可雙擊「長度」欄位，直接進入該場次的既有編輯模式並聚焦長度欄位。

## 採用方案

沿用現有儲存佇列與整列編輯介面，只補強切換場次的事件順序及長度欄位入口。避免改成每次按鍵都呼叫 API，也不重寫整套表格狀態。

## 行為設計

### 自動儲存

- 點擊目前編輯列以外的操作時，使用最新的 draft 快照排入儲存佇列。
- 點擊另一場「編輯」時，明確等待目前場次儲存完成，再載入下一場資料。
- 若儲存失敗，沿用既有錯誤提示，並保留目前編輯內容，不切換到下一場。
- 「取消」按鈕與 `Esc` 保留放棄本次修改的語意，不觸發儲存。
- `Enter` 與原本「儲存」按鈕行為維持不變。

### 批次日期

- 電腦版與手機版的批次日期狀態，初始值改為台北當日日期。
- 切換集數或完成一次批次日期修改後，日期恢復為當日日期。
- 「清除日期」按鈕維持原本功能。

### 電腦版雙擊長度

- 只有非編輯狀態下的「長度」儲存格加入雙擊行為。
- 雙擊後進入目前既有的整列編輯模式，並自動聚焦及選取長度內容，方便直接輸入。
- 其他儲存格不加入雙擊功能。
- 手機版卡片與 Bottom Sheet 不做任何調整。

## 修改範圍

- 主要修改 `src/components/SceneTable.tsx`。
- 不修改 Supabase schema、資料格式、批次更新 API 或手機版版面。
- 若測試需要可測的事件協調邏輯，只抽出最小 helper，不做無關重構。

## 驗證條件

1. 編輯場次 A 後直接點場次 B 的「編輯」，A 儲存成功且 B 正常進入編輯。
2. 編輯後點頁面其他操作，重新整理仍看得到修改內容。
3. 點「取消」或按 `Esc`，本次未儲存內容不寫入。
4. 批次日期在電腦版與手機版預設顯示今日，完成批次修改後仍回到今日。
5. 電腦版雙擊長度欄位會進入編輯並聚焦長度；雙擊其他欄位無反應。
6. 手機版編輯流程沒有行為或版面變更。
7. `npm run lint` 與 `npm run build` 通過，再用 Codex app 內部 Browser 驗證電腦版互動。
