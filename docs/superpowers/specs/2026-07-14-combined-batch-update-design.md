# 場次狀態與日期合併批次修改設計

日期：2026-07-14（Asia/Taipei）

## 目標

讓使用者在電腦版與手機版選取多個場次後，可以在同一次操作中修改狀態、日期，或同時修改兩者，並只送出一次批次寫入。

## 採用方案

採用共用的「批次修改」視窗，取代目前分開送出的「批次修改狀態」與「批次修改日期」。電腦版與手機版共用相同資料與送出邏輯，只依螢幕尺寸調整視窗呈現方式。

這個方案讓每個欄位都有明確的「維持不變」選項，可避免使用者只想改狀態時，意外把日期改成今日。

## 操作介面

### 批次工具列

- 選取一個以上場次後，顯示「批次修改（N）」按鈕。
- 「批次刪除」維持獨立按鈕與既有確認流程，不混入一般批次修改視窗。
- 電腦版與手機版都從「批次修改（N）」進入同一套設定。

### 批次修改視窗

狀態欄位提供：

- 維持不變，預設值。
- 已初剪。
- 已精剪。
- 整場刪除。
- 清除狀態。

日期欄位提供三種模式：

- 維持不變，預設值。
- 指定日期；選取後顯示日期輸入框，預設為台北當日日期。
- 清除日期。

送出規則：

- 狀態與日期至少一項不是「維持不變」，「套用到 N 個場次」才可按。
- 使用者可以只改狀態、只改日期，或同時修改兩者。
- 按下套用後，確認訊息列出場次數量與實際會變更的欄位。
- 確認後只送出一次批次更新。

### 響應式呈現

- 電腦版使用置中的對話視窗。
- 手機版沿用現有窄螢幕操作語言，使用底部展開的 Bottom Sheet。
- 兩種版面共用同一份狀態、驗證與送出 handler，不各自維護功能邏輯。

## 資料流

1. `SceneTable` 保存已選場次，以及批次視窗中的狀態與日期設定。
2. 使用者確認後，`SceneTable` 只把已選場次索引與「實際要改的欄位」傳給父層。
3. `EpisodeDetail` 將 patch 合併到每個目標場次的既有完整資料。
4. `EpisodeDetail` 呼叫既有 `batchUpdateScenes` 一次，把所有目標場次寫入 Supabase。
5. 寫入成功後，本機 cache 與 summary 各同步一次。

不修改 Supabase schema，也不新增第二套批次 API。

## 成功、取消與錯誤行為

- 成功：關閉視窗、清除場次選取，並把批次設定重設為「維持不變」。
- 關閉視窗或按 `Esc`：不送出任何寫入，保留場次選取，並把批次設定重設為「維持不變」。
- 在瀏覽器確認訊息選擇取消：回到批次視窗，保留場次選取與批次設定。
- 寫入失敗：沿用既有錯誤提示，保留視窗、場次選取與使用者設定，方便直接重試。
- 寫入期間：停用輸入與送出按鈕，避免重複提交。
- 確認訊息與實際 patch 必須由同一份設定產生，避免畫面說一套、送出另一套。

## 修改範圍

- `src/components/SceneTable.tsx`：整合電腦版與手機版批次入口、視窗與互動狀態。
- `src/components/EpisodeDetail.tsx`：把狀態與日期合併成單一批次更新 handler。
- 若需要可測的純邏輯，只抽出一個最小 helper，負責驗證設定、建立 patch 與確認摘要。
- 新增聚焦於合併批次行為的測試。

## 不在範圍內

- 不修改場次單筆編輯、自動儲存或選取方式。
- 不修改批次刪除與批次匯入。
- 不修改狀態選項、日期格式、資料庫 schema 或權限規則。
- 不順手重構其他表格或手機版元件。

## 驗收條件

1. 電腦版可只修改所選場次的狀態，日期保持原值。
2. 電腦版可只修改所選場次的日期，狀態保持原值。
3. 電腦版可在一次確認、一次批次寫入中同時修改狀態與日期。
4. 手機版通過以上三種操作，行為與電腦版一致。
5. 「清除狀態」只清除狀態；「清除日期」只清除日期；兩者也可同時清除。
6. 狀態與日期皆為「維持不變」時不可送出。
7. 取消不寫入且保留已選場次；失敗保留設定與選取，成功才清除選取。
8. 同時修改時，底層 `batchUpdateScenes` 只呼叫一次，cache 與 summary 各更新一次。
9. 既有批次刪除、Shift 選取、場次編輯流程維持原行為。
10. 聚焦測試、`npm run lint`、`npm run build` 全部通過，並用 Codex app 內部 Browser 實際驗證電腦版與手機版操作。
